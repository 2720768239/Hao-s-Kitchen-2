# AGENT.md

适用范围：仓库根目录及其全部子目录。

## 1. 项目定位

本项目是 `极饿世代` 的 V1 实现，一个基于 `Next.js App Router + TypeScript + SQLite` 的家庭饭局点菜系统，当前范围包括：

- 前台 invite 单页点菜
- 3 分钟菜品占位
- 主厨工具台
- 菜单管理
- 历史饭局
- 图片上传
- SSE 实时刷新

不要把它当成通用外卖平台，也不要擅自扩展到 AI 推荐、多餐厅、多角色后台或复杂账号体系。

## 3. 关键业务规则

- `/` 固定显示关闭态；只有有效 `/invite/[token]` 才进入前台点菜页。
- 同一道菜在同一时间只能有一个有效占位。
- 同一道菜在同一饭局中只能被正式提交一次。
- 占位有效期固定为 `3 分钟`，过期自动释放。
- 主厨点击 `英雄集结` 创建新饭局和新的 invite token。
- 主厨点击 `群雄归隐` 归档当前饭局、清理未提交占位，并让旧 invite 失效。
- 前台提交成功后停留当前页，用轻提示反馈，不跳独立成功页。
- 实时同步目前统一走 SSE，收到事件后客户端执行刷新，不维护复杂前端共享状态。

如果改动会触碰这些规则，必须同步检查领域逻辑、API、UI 和测试。

## 4. 技术栈与目录边界

- `src/app`：页面与 API 路由
- `src/components`：前台与主厨端组件
- `src/server`：服务层、查询组合、业务逻辑、事件总线
- `src/db`：SQLite 客户端、迁移、seed、schema
- `src/lib/auth`：主厨鉴权与 session
- `src/lib/domain`：领域常量、错误、类型
- `tests/domain`：纯领域规则测试
- `tests/api`：路由/接口测试
- `tests/ui`：组件与页面行为测试
- `tests/e2e`：Playwright 端到端测试

优先保持现有分层，不要把复杂业务判断直接堆进路由处理函数或 React 组件。

## 5. 代码约定

- 使用 TypeScript 严格模式，复用现有 `@/*` 路径别名。
- 保持现有风格：ESM import、双引号、分号、尽量显式类型。
- API 路由保持薄层：做鉴权、参数校验、错误映射，把业务交给 `src/server`。
- 输入校验优先使用 `zod`。
- 可预期业务错误统一走 `DomainError` / `ConflictError`，不要直接返回散乱字符串。
- 数据库访问延续当前模式：`better-sqlite3` + `drizzle` schema，复杂事务放在服务层。
- 前端在既有设计语言内改动，避免引入与当前产品气质不一致的视觉方向。
- 不要随意新增重量级依赖；能复用现有能力就不要扩栈。

## 6. 测试约定

常用命令：

```bash
npm test
npm run lint
npm run build
npm run test:e2e
```

数据库相关初始化：

```bash
npm run db:migrate
npm run db:seed
```

测试时遵循现有模式：

- 路由测试优先复用各 route 文件导出的 `set...TestDatabase(...)`
- 测试数据库使用内存库，不要依赖真实 `data/` 目录
- 改领域规则时至少补 `tests/domain`
- 改接口行为时补 `tests/api`
- 改页面交互时补 `tests/ui`
- 只有在跨页面主流程、SSE、登录态或真实浏览器行为受影响时，再跑或补 `tests/e2e`

做改动后，至少运行与改动直接相关的测试；在可行时补跑 `npm test`、`npm run lint`、`npm run build`。

## 7. 数据与部署注意事项

- 默认数据库路径：`data/hao-kitchen.sqlite`
- 默认上传目录：`data/uploads`
- Docker 启动时执行 `db:migrate` 和 `start`，不会自动反复 seed
- `db:seed` 只用于空库初始化默认菜单，不应覆盖已有业务数据
- 上传文件目前仅支持 `PNG / JPG / WEBP`，大小不超过 `5MB`

不要在开发任务中随意修改 `data/` 下真实数据，除非用户明确要求。

## 9. Agent 工作方式

- 先理解当前改动所在层级，再动手。
- 优先做小而完整的改动，避免无关重构。
- 保持业务规则、命名和文案一致性。

如果只是补文档或小修复，也应尽量让测试与文档保持同步。
