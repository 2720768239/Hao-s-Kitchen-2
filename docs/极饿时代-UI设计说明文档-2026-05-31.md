# 极饿时代 UI 设计说明文档
版本：V1 设计说明  
日期：2026-05-31

## 1. 文档目的

本文档用于沉淀 `极饿时代` 第一版的页面 UI 设计规则、页面结构、组件状态、文案落点与视觉约束，作为设计定稿、前端实现与后续迭代的统一依据。

本文档基于以下内容整理：

- `完整需求文档`
- `项目开发文档`
- `命名与文案清单`
- 视觉参考图 `docs/ui风格.png`
- 本轮对话中已经确认的产品与视觉调整

## 2. 本轮确认后的 UI 修正口径

以下两条以本文档为准，优先级高于早期草稿或旧讨论：

- 页面中不使用辣椒图形、辣椒贴纸、辣椒角标、辣椒印章或任何辣椒形象装饰
- 后台不做桌面 Web 管理台，第一版 UI 统一按 `移动端主厨工具台` 设计

## 3. 设计目标

第一版 UI 需要同时满足两件事：

1. 前台要有强烈的世界观和海报感，让人一进来就知道这不是普通点单页
2. 交互要足够熟悉和直接，不能因为风格太满而影响看菜、选菜、提交

换句话说：

- 前台负责“有气质、有情绪、有记忆点”
- 后台负责“拿起手机就能管、就能看、就能改”

## 4. 整体视觉方向

## 4.1 关键词

- 海报感
- 纸张肌理
- 黑红米白主色
- 刷字标题
- 贴纸感与盖章感
- 江湖饭局气质
- 强主厨人格

## 4.2 视觉气质原则

- 不做电商感
- 不做传统餐厅菜单感
- 不做中后台系统灰色面板感
- 不做二次元、赛博或潮玩方向
- 不做辣味主题化图形表达

它更像：

- 一张被做成产品的厨房海报
- 一套有世界观的家庭饭局点菜工具

## 4.3 色彩建议

建议以 4 组主色完成第一版：

- 主黑：用于大标题、核心按钮、结构边界
- 米白：作为主要背景色，承担纸张感
- 深红：用于状态强调、印章、重点标签、主厨语气强化
- 灰褐：用于辅助信息、边界线、说明文字

建议避免：

- 过亮的纯红大面积铺底
- 高饱和多色混搭
- 绿色、蓝色等会打散世界观的功能色做主视觉

## 4.4 材质与纹理

- 页面背景可使用轻纸张纹理、旧报纸感噪点、轻微磨损边缘
- 卡片和弹层可以有印刷品、贴纸、便签、票据感
- 所有纹理都应轻量控制，不能影响内容识别

## 4.5 字体与排版建议

- 主标题使用偏书写感、刷字感或具有张力的中文展示字体
- 正文与交互信息使用清晰的无衬线字体
- 重要状态词可放大、压粗、加盖章处理
- 避免整页都用手写体，保证可读性

排版上建议：

- 大标题强对比
- 文案留白明显
- 菜卡信息分层清楚
- 用大块面和短句节奏组织页面，不做密集说明书式排版

## 5. 设计系统基础规则

## 5.1 圆角与边框

- 卡片圆角建议偏大，营造贴纸和票据拼贴感
- 边框可以使用较粗描边，增强海报化结构
- 重点区域可叠加手工裁切感或不完全规则边缘，但要克制

## 5.2 阴影与层级

- 阴影不追求现代悬浮卡片效果
- 更适合做“纸片叠压”层次
- 弹层与底部抽屉可以比普通卡片更厚重，形成“压上来”的舞台感

## 5.3 图标原则

- 图标尽量少
- 能用文字表达的优先用文字
- 若必须使用图标，风格应偏简洁、粗线条、带一点印章感
- 禁用辣椒图标

## 5.4 图片原则

- 菜图应该真实、诱人、构图直接
- 不建议过度滤镜化
- 菜图与页面风格的冲突通过外框、贴纸、文案解决，而不是把菜图本身做旧

## 6. 前台 H5 设计说明

前台是沉浸式单页，逻辑结构固定为：

1. 开场区
2. 菜单区
3. 底部固定操作区

## 6.1 页面一：开场态

### 目标

- 建立世界观
- 告知当前营业状态
- 引导进入菜单浏览

### 必备内容

- 主标题：`极饿时代`
- 一句随机主厨宣言
- 当前状态：`英雄集结` 或 `群雄归隐`
- 向下浏览引导

### 视觉要求

- 标题必须是全屏第一视觉
- 状态词要有明显仪式感，可做成盖章或横幅
- 背景允许有轻纹理、贴纸、噪点、拼贴边角
- 不出现辣椒元素

### 文案气质

- 有态度
- 不像客服
- 不像营销 banner
- 不要太用力耍狠，保留一点幽默

## 6.2 页面二：菜单主视图

### 目标

- 让用户快速看懂今晚能点什么
- 同时感知哪些菜已经被人盯上或拿下

### 菜卡结构

每张菜卡建议包含：

- 菜名
- 菜图
- 一句短描述或主厨短评
- 标签
- 操作按钮
- 当前状态提示
- 已点人信息

### 菜卡状态

#### `available`

- 操作按钮可点击
- 按钮文案可使用统一主文案或随机短句
- 推荐优先统一用一个主口径，避免第一版过于花

#### `holding`

- 显示：`有人先盯上了`
- 按钮禁用
- 样式变灰或降低饱和度，但仍需可读

#### `claimed`

- 显示：`已被 XX 拿下`
- 操作按钮不可用
- 点菜人信息必须清晰可见

### 布局建议

- 单列大卡或双列偏大卡都可，但第一版更推荐单列
- 单列更利于做海报感、容纳文案，也更适合移动端识别状态

## 6.3 页面三：底部固定操作区

这是前台最重要的交互组件，用于合并原本分散的：

- 共享已点信息
- 我的已选信息
- 报名提交入口

### 固定条结构

- 左侧：清单入口与数量提示
- 中间：辅助文案 `馋好了就报上名来`
- 右侧：主按钮 `馋`

### 视觉要求

- 常驻底部
- 视觉重量足够，不能像普通浅灰工具条
- 更像一张贴在页面底部的操作票据或悬浮招牌

## 6.4 页面四：底部展开态

### 目标

- 让用户看到当前已点全局
- 明确区分自己能操作的内容和只能看的内容

### 结构建议

展开后建议分为 2 组：

- `你已经馋上的`
- `今夜已被别人抢走的`

### 每条记录建议展示

- 菜名
- 点菜人
- 状态标签
- 仅自己的临时占位显示移除操作

### 交互规则

- 只能移除自己的临时占位菜
- 别人已点或别人占位的菜不可删
- 展开态不跳新页面，使用抽屉或底部面板

## 6.5 页面五：提交态

### 目标

- 在不打断单页体验的前提下完成报名字和备注

### 表单字段

- 名字
- 备注

### 主操作

- 主按钮文案：`馋`
- 页面辅助文案：`馋好了就报上名来`

### 交互形式

- 推荐使用底部弹层或抽屉
- 不建议跳转到独立表单页

### 视觉要求

- 字段数量少
- 视觉上像“留名”而不是“下单结算”
- 输入区仍保持世界观语气，但不能影响理解

## 6.6 页面六：提交成功态

### 目标

- 给用户明确反馈
- 保留单页叙事的完整性

### 建议内容

- 主文案：`你饿完了`
- 副文案：`我去厨房了`
- 已提交菜品简要回看

### 设计要求

- 成功反馈要有仪式感
- 不做电商式“下单成功”
- 仍然停留在前台单页语境里

## 6.7 页面七：群雄归隐态

### 目标

- 在打烊时保持世界观完整
- 清楚表达当前不能继续点菜

### 主文案

- `群雄归隐`

### 呈现方式

- 保留开场视觉框架
- 隐藏或禁用菜单操作与提交能力
- 不要只剩一句冰冷提示

### 补充要求

- 文案可以少，但气氛要足
- 不使用“打烊”“结束”“暂不营业”等普通商业化提示作为主标题

## 7. 后台移动端设计说明

后台第一版是 `移动端主厨工具台`，不是桌面后台。

整体要求：

- 同一世界观
- 明显更工具化
- 单手操作友好
- 信息优先级高于装饰性

## 7.1 页面一：主厨首页

### 目标

- 主厨一打开就知道现在是什么状态
- 能快速切换 `英雄集结 / 群雄归隐`
- 能快速进入今晚待做内容

### 首页内容

- 页面标题
- 当前营业状态卡
- 状态切换按钮
- 今晚待做数量概览
- 进入菜单管理入口

### 视觉要求

- 状态卡必须成为首页第一焦点
- 比前台更干净，但仍保留纸张、贴纸、印章语汇

## 7.2 页面二：待做清单

### 目标

- 让主厨一眼知道今晚要做什么

### 信息结构

每条记录至少包含：

- 菜名
- 谁点的
- 备注
- 提交时间

### 布局建议

- 按列表展示，优先可读性
- 一屏内尽量多看几条，不做大面积无效装饰
- 菜名优先级最高，点菜人和备注次之

### 空状态

- 允许使用世界观文案
- 但要让主厨立刻明白“当前没人点菜”

## 7.3 页面三：菜单管理

### 目标

- 快速维护今晚可点菜品

### 列表项建议包含

- 菜图缩略图
- 菜名
- 标签或简述
- 上架/下架状态
- 编辑入口
- 排序入口或排序指示

### 设计原则

- 更像工具列表，不像内容运营平台
- 操作清晰，减少隐藏层级

## 7.4 页面四：新增 / 编辑菜品

### 字段建议

- 菜名
- 菜图
- 描述
- 标签
- 是否上架
- 排序值

### 交互形式

- 移动端可使用全屏编辑页或底部弹层
- 如果字段较多，优先使用全屏编辑页，避免弹层过挤

### 保存反馈

- 保存成功要有明确提示
- 不需要复杂动画

## 8. 核心组件说明

## 8.1 状态标签

统一用于：

- `英雄集结`
- `群雄归隐`
- `有人先盯上了`
- `已被 XX 拿下`

设计要求：

- 有足够识别度
- 可做印章、票签、横贴条样式
- 同一套组件语言贯穿前后台

## 8.2 主按钮

前台主按钮固定为：

- `馋`

设计要求：

- 足够短促有劲
- 一眼能认出是主操作
- 不要做成普通圆角蓝按钮风格

## 8.3 菜卡

设计要同时承载：

- 图片
- 名称
- 短评
- 标签
- 状态
- 操作

因此必须重视信息层级，避免所有元素都抢戏。

## 8.4 底部抽屉

这是前台最关键的复合组件。

要点：

- 展开与收起反馈明显
- 列表滚动与页面滚动不打架
- 关闭路径明确
- 在小屏设备上仍能完整看到主操作按钮

## 8.5 输入组件

- 样式可做世界观包装
- 但输入框边界、标签、placeholder 必须清楚
- 不使用过重背景纹理影响阅读

## 9. 交互动效建议

第一版动效只做轻量增强，不做复杂炫技。

建议保留的动效：

- 菜加入临时占位时的轻反馈
- 底部抽屉展开 / 收起
- 状态切换时的轻微过渡
- 提交成功后的确认反馈

建议避免：

- 大面积视差
- 长时动画
- 高频抖动
- 会拖慢 H5 的粒子或复杂特效

## 10. 响应式与适配建议

## 10.1 前台

- 以手机竖屏为主设计场景
- 重点适配常见 iPhone 和安卓主流宽度
- 确保底部固定区不会遮挡关键内容

## 10.2 后台

- 同样以手机竖屏为主
- 主厨常用场景应假设为单手快速查看和切换

## 10.3 安全区域

- 注意顶部状态栏和底部 Home Indicator 安全区
- 底部固定操作条、底部抽屉都必须处理安全区留白

## 11. 文案落地原则

- 前台文案可以更有戏
- 后台文案要更直接
- 状态词固定，不要一页一个叫法
- 第一版尽量少做随机文案池，优先保证一致性

必须固定的关键文案：

- `英雄集结`
- `群雄归隐`
- `馋`
- `馋好了就报上名来`
- `你饿完了`
- `我去厨房了`

## 12. 视觉禁用项

第一版明确禁止以下表达：

- 辣椒图标
- 辣椒插画
- 辣椒角标
- 辣椒贴纸
- 辣味主题拟物图形
- 电商购物车图标作为主视觉核心
- AI 推荐区块视觉入口
- 支付、结算、金额类界面语义

## 13. 设计交付建议

设计稿交付时建议至少包含：

- 前台开场态
- 前台菜单态
- 前台底部展开态
- 前台提交态
- 前台成功态
- 前台群雄归隐态
- 后台移动端首页
- 后台待做清单页
- 后台菜单管理页
- 后台编辑菜品页

同时建议附带：

- 色板
- 字体说明
- 主按钮 / 次按钮规范
- 状态标签规范
- 菜卡规范
- 底部抽屉规范
- 表单组件规范

## 14. 开发对接注意事项

- 视觉风格可以强，但开发落地必须优先保证信息可读与状态可辨
- 前台底部固定区是最关键交互，请优先高保真还原
- `holding` 与 `claimed` 的视觉差异必须明显
- 后台移动端应避免套用传统 PC 后台组件思路

## 15. 第一版 UI 验收标准

满足以下条件时，可认为第一版 UI 设计达到可开发状态：

1. 前后台页面结构都已明确
2. 核心状态视觉已统一
3. 前台底部固定操作区交互已清楚表达
4. `英雄集结 / 群雄归隐` 视觉语义稳定
5. 菜卡三态可直观看懂
6. 提交态与成功态不再带有传统下单语义
7. 后台移动端页面可直接支撑主厨操作
8. 全套页面中不存在辣椒图形装饰

## 16. 结论

`极饿时代` 第一版 UI 的正确方向，不是做成常规点餐产品，也不是做成一套过度概念化海报，而是做成：

`一个有强主厨人格、强海报气质，但交互非常清楚的家庭聚餐共享点菜工具。`

前台负责氛围和记忆点，后台负责效率和掌控感。只要这条线守住，第一版就会既有辨识度，也真能落地。
