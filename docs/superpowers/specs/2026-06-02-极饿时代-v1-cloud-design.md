# 极饿时代 V1 云服务器实现规格

日期：2026-06-02

## 1. 目标

实现可部署到阿里云单实例服务器的完整 V1：

- 客人通过饭局邀请链接进入前台 H5
- 主厨通过受口令保护的后台管理饭局、菜单和待做清单
- 服务端可靠处理临时占位、重复点菜、饭局归档和状态同步
- SQLite 与上传图片保存在 Docker 持久化目录

前台视觉以 `docs/image-03.png` 为主基准，后台视觉以 `docs/image-02.png` 为主基准。

## 2. 范围

### 2.1 本轮包含

- 前台沉浸式单页
- 主厨后台登录
- `英雄集结 / 群雄归隐` 状态操作
- 每场饭局的独立邀请链接
- 菜单浏览、图片上传、增改、上下架和排序
- `3` 分钟临时占位
- 填名和备注提交
- SSE 变更通知
- 待做清单
- 历史饭局列表与只读详情
- Docker Compose 部署
- Nginx HTTPS 示例
- SQLite 与上传图片的手动备份命令

### 2.2 本轮不包含

- 完整账号体系
- 多主厨协作
- 多实例部署
- PostgreSQL
- 自动证书申请
- 自动定时备份
- AI 推荐
- 支付

## 3. 技术架构

采用单体 `Next.js` 应用：

- Web 页面、API、SSE、上传处理运行在同一个应用容器
- 使用 TypeScript
- 使用 Drizzle ORM 管理 SQLite
- 使用 Docker Compose 管理应用和持久化挂载

主要路由：

- `/`：统一展示 `群雄归隐`
- `/invite/:token`：客人前台 H5
- `/chef/login`：主厨口令登录
- `/chef`：主厨首页
- `/chef/to-cook`：待做清单
- `/chef/dishes`：菜单管理
- `/chef/history`：历史饭局

## 4. 饭局生命周期

饭局持久状态只有：

- `gathering`
- `archived`

后台页面不设置“开启新饭局”按钮。状态卡直接提供：

- `英雄集结`
- `群雄归隐`

规则：

1. 当前没有 `gathering` 饭局时，主厨点击 `英雄集结`，服务端创建新饭局并生成高熵邀请 token。
2. 当前存在 `gathering` 饭局时，后台显示当前邀请链接。
3. 主厨点击 `群雄归隐`，服务端归档当前饭局、删除未提交占位并使邀请链接永久失效。
4. 已归档饭局不可重新打开。
5. 再次点击 `英雄集结` 会创建下一场新饭局和新邀请链接。

邀请链接不设置固定时间期限。它只在对应饭局归档时失效。

## 5. 前台访问规则

只有当前 `gathering` 饭局的有效邀请链接可以展示菜单。

以下情况统一展示 `群雄归隐` 页面：

- 直接访问 `/`
- 缺少 token
- token 不存在
- token 对应饭局已归档
- 主厨刚刚点击 `群雄归隐`

归隐页保留完整世界观视觉，不显示菜单、占位和提交入口。

## 6. 前台页面与交互

前台以 `docs/image-03.png` 为视觉主基准，使用移动端单页长页：

1. 海报式开场区
2. 单列菜单
3. 底部固定操作条
4. 底部抽屉
5. 留名提交弹层
6. 提交成功反馈
7. `群雄归隐` 页面

菜品状态：

- `available`：可以占位
- `holding`：已有有效临时占位，不可再选
- `claimed`：已经正式提交，显示点菜人

底部抽屉固定分组：

- `你已经馋上的`
- `已有馋主`

用户只能移除属于自己的临时占位。提交弹层只包含名字和备注。浏览器保存最近一次名字，方便同一客人再次填写。

后台以 `docs/image-02.png` 为视觉主基准，并保持移动端优先。

## 9. 一致性规则

SQLite 单实例写操作使用事务串行化处理：

1. 创建占位前，删除已经过期的占位。
2. 在事务中检查菜品是否已经正式提交。
3. 在事务中检查菜品是否存在有效占位。
4. 只有第一个成功写入者获得占位。
5. 提交时再次校验占位属于当前客户端且尚未过期。
6. 提交成功后写入订单，并删除对应占位。
7. 同一饭局、同一道菜只能存在一条正式结果。
8. 归档饭局时删除全部未提交占位。

数据库约束与服务端事务同时兜底，前端禁用状态只用于及时反馈。

## 10. SSE 同步

SSE 只负责通知“状态已经变化”，不直接承担复杂状态合并。

推送范围：

- 占位创建或撤回
- 占位过期
- 正式提交
- 菜单新增、编辑、排序或上下架
- 饭局归档

浏览器断线后自动重连。连接建立或恢复后，客户端立即重新读取完整状态。

## 11. 鉴权与上传

### 11.1 主厨后台

- 使用单一主厨口令
- 口令以哈希形式保存在环境变量或首次初始化配置中
- 登录成功后创建随机会话 token
- 浏览器使用 `HttpOnly`、`SameSite` Cookie 保存会话
- 生产环境 HTTPS 下设置 `Secure`

### 11.2 邀请链接

- 邀请 token 是进入当前饭局的能力凭证
- token 使用密码学安全随机数生成
- 归档后永久失效

### 11.3 图片上传

- 仅允许配置的图片 MIME 类型和扩展名
- 限制单文件大小
- 使用服务端生成的文件名
- 文件写入挂载的上传目录
- 数据库只保存相对路径

## 13. 异常处理

- 抢菜失败：提示“这道菜有人先盯上了”，并刷新菜单
- 占位过期：自动释放；提交失败时明确列出失效菜品
- SSE 断线：自动重连并完整刷新
- 上传失败：保留编辑表单并显示原因
- 主厨口令错误：不创建会话
- token 无效或饭局归档：统一显示 `群雄归隐`

## 16. 验收标准

1. 无有效邀请链接时只能看到 `群雄归隐`
2. `英雄集结` 创建新饭局和新邀请链接
3. `群雄归隐` 归档当前饭局，旧链接永久失效
4. 已归档饭局不可重新打开
5. 同一道菜不会被重复占位或重复正式提交
6. 占位超过 `3` 分钟后可靠释放
7. SSE 让前后台及时感知变化
8. 主厨后台受口令保护
9. 菜单图片可上传到持久化目录
10. 历史饭局可只读查看
11. Docker Compose 可在阿里云单实例服务器运行
12. 提供 Nginx HTTPS 示例和手动备份方式
